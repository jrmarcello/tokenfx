# Next.js App Router Conventions

## Server vs Client Components

- Server Components by default. Add `'use client'` only when:
  - Event handlers (`onClick`, `onChange`, ...)
  - Browser APIs (`window`, `document`, `localStorage`)
  - React hooks that require a client runtime (`useState`, `useEffect`, `useReducer`, refs with imperative usage)
  - Third-party libraries that require the client (most chart libs, some UI libs)

## Data Fetching

- Server Components call `lib/queries/*` directly — do not fetch your own API routes from the server.
- Pass data down as props to Client Components.
- For revalidation on a schedule, use `export const revalidate = N` or `fetch(..., { next: { revalidate: N }})`.

## Mutations

- Prefer **Server Actions** for form submits (built-in CSRF protection, colocated with component).
- Use **Route Handlers** (`app/api/*/route.ts`) for:
  - External triggers (CLI scripts, cron, webhooks)
  - Non-form JSON APIs consumed by client code
- After every mutation that changes data the UI reads, call `revalidatePath(...)` or `revalidateTag(...)`.

## Route Segments

- Provide `loading.tsx` for any route that performs meaningful async work.
- Provide `error.tsx` (Client Component) for routes that can fail — it isolates the error boundary.
- `layout.tsx` should not fetch data that varies per page unless it's truly shared.

## Components

- shadcn/ui primitives live in `components/ui/` (generated by the CLI; customize in-place).
- Domain-specific components colocated near their usage under `components/<domain>/` or within the route segment.
- Client Components should be leaves where possible — keep the Server/Client boundary as deep as it can go.

## Data Layer from UI

- No direct `better-sqlite3` imports from Client Components. Queries are server-only.
- If a client needs server data mid-session, route through a Server Action or Route Handler.
